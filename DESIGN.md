## What it is

**Unmodified SQLite, multiple concurrent writers, constraints intact, coordinated by a small lease service instead of a consensus cluster — with E2EE nearly free because the server never interprets values.**

Every writer holds a full or partial local replica; reads are function calls, writes are locally-committed SQLite transactions (microseconds) validated authoritatively via held leases, shipped asynchronously as logical row changes. **The architecture in one line: a coherent write-back cache protocol for relational data — MESI for databases.** Everyone solved read-side caching (Turso, PowerSync, ElectricSQL replicate the authority's reads down); write-side caching is unsolved except via consensus RTTs or optimism-with-merges. Sticky ownership is the third option. Turso runs SQL at the center and replicates outward; homebase runs SQL at the edge and coordinates upward.

**Why E2EE is a moat, not a feature:** server-side query execution requires server-side plaintext (uniqueness needs equality, WHERE needs reading, indexes need order). Therefore E2EE ⇒ client-side execution ⇒ client-side enforcement ⇒ something lease-shaped. Authority-model systems cannot follow even in principle.

**Honest scope:** online-preferred with explicit bounded offline for mutable SQL (pin API: declared-intent long leases; sticky residue ≥ TTL−heartbeat for surprise drops); offline-first for append-shaped data (ledger pattern). Conceded openly: hot-contention rows resolve in seconds (lease migration) vs. Postgres's ~1ms lock queue — queue-like workloads are documented anti-patterns; partial replication alone is table stakes (others have it); durability is device-then-space (Litestream contract, per-table sync-commit opt-out, watermark API exposed).

**ACID label:** atomic (batch = transaction; torn batches impossible); consistent for all declared constraints globally (write skew on undeclared invariants at default isolation — same hole as Postgres defaults; `BEGIN IMMEDIATE` escalates); **serializable writes, three-tier reads** — local (0 RTT, per-shape freshness), owned (0 RTT, authoritative), consistent (1 RTT, true snapshot cut); durable per watermark.

## homebase — the kernel (frozen shape)

**The kernel in one line: an opaque KV server with read/write prefix leases.** One ordered map per space — tuple keys (`Vec<Bytes>`, component-wise prefix, order-preserving flat encoding: 0x00-escape + two-byte terminator, so tuple-prefix ⟺ encoded-byte-prefix and prefix scans are plain byte-range scans) → opaque values, tagged `(device, device_seq, epoch, ver, admission_seq)` — plus a lease table, registered subscriptions, and **per-prefix aggregates maintained at write time** (max admission seq + live-key count per `(depth, prefix)`, updated along each written key's prefix path in the same atomic batch — ≤ max-components extra records per key): the durable form of the augmented range-max tree, answering "anything new under this prefix since my cursor?" and "how many live keys?" without scans. Seven verbs:

```
acquire/renew/release   # mode read|write. TTL, batch acquire → barrier seq,
                        # contention piggybacks on renewal, steal denied pre-deadline
                        # (unless granted stealable and acquire passes steal=true)
put_batch               # atomic; every key covered by valid write lease; epoch-fenced;
                        # per-key ver monotonicity enforced; tombstone deletes
get / list
read_at(ranges, cursors) -> (S, Δ)   # atomic consistent cut at one admission point;
                                     # also the delta feed that drives shapes
```

Prefix-scoped tokens enforced on reads AND writes. **Two lease modes:** write excludes everything; read coexists with read and excludes write. Read leases guard read sets (FK parents take read; child inserts take write on per-child guard keys; ledger appends = long write on device-scoped prefixes). **Stealable leases:** a lease may be marked `stealable` at acquire; a later `acquire(steal=true)` over the same prefix preempts it immediately, pre-deadline — the fresh grant's epoch fences the old holder (its next put/renew fails; correctness never depends on it noticing). Leases not marked stealable keep strict pre-deadline denial. This is the single-active-device primitive: WhatsApp-style "one device active at a time" = one stealable write lease on the account prefix; activating a new device steals it, deactivating the old one at that instant. Invariants: no write admission without covering valid write lease · no incompatible overlap · no read→write upgrade, ever · epochs are correctness, timestamps availability (asymmetric expiry) · strict local expiry · barrier = serializability · demand-driven stickiness, min-hold ≈ 2× heartbeat · TTL policy: kernel cap → class defaults (witness ~10s non-sticky, row ~5m, db ~1h) → app pin.

**Deployment shape:** a server owns **shards** of spaces; shard = one slatedb instance (object-store-native LSM: local-NVMe object store for dev/self-host, S3 for prod) = one WAL, group-committing across its spaces (storage key = space id ⊕ kind ⊕ encoded key; per-space records: data with inline tags, changelog keyed by latest-change admission seq, lease by-id + by-prefix indexes, counters). **Each space is an actor**: one mailbox, one logical thread; verbs for a space are dequeued and run strictly one at a time (`now` stamped at dequeue — the linearization point that admission seqs, epoch monotonicity, and the deterministic-sim story all rely on); concurrency exists only across spaces. Actors are runtime machinery, not identity: spun up lazily on a space's first verb after server start, parkable when idle (all state is in the store; an actor restart is a mailbox restart).

**Tenancy & space sizing.** A space is the unit of every guarantee — one admission sequence, one lease table, one `read_at` cut, one actor; nothing crosses spaces. Sizing rule: everything that must ever be transactionally consistent together shares a space, nothing else should. In practice: **space ≈ one logical database plus its device-local satellites** (per-device ledger prefixes, the account prefix carrying the single-active-device stealable lease) — one SQL db is the floor (the witness compiler's barrier works within one space), unrelated dbs are excluded (they'd serialize through the same actor and inflate the changelog/aggregate domains for nothing). Above spaces sits the **tenant** (uuid): the administrative quantum — keys, quotas, billing. The kernel stays tenant-oblivious (SpaceId is already globally unique; tenant never appears in storage keys or verbs); tenant lives in the directory (tenant → owned spaces, space → shard placement) and in tokens (tenant keys mint space-scoped claims). Plane split: tenant creation/keys/quotas = **control plane** (rare, administrative); space creation = **tenant plane** — cheap, token-authorized, quota-checked, done constantly by apps (per-user/per-document dbs; thousands of spaces per tenant). Creating a space is a directory entry (owner + placement); the shard does no work until the first verb lazily spins the actor.

**Directory & metering (tentative — deferred until the commercial tier; finalize then).** Multi-tenancy, the directory, and metering are *not* needed for the OSS/single-operator milestones — nothing below depends on them, and the kernel is tenant-oblivious either way. Sketch to finalize later: tenant identity is never a request parameter — it is *derived* from the token's signing key (key id → tenant record → verify signature) and cross-checked against the directory: request admitted iff the directory says the token's space belongs to the signing tenant. The directory is a dedicated system store reusing the kernel's own machinery (ordered store, tuple keys, record kinds): `(Tenant, id)` → keys/quotas/status/created; `(SpaceDir, space_id)` → owning tenant, created_at, creator, shard placement, status; `(TenantSpaces, tenant, space)` → marker (reverse index = space listing + the quota check `create_space` performs). `create_space(tenant_token)` = capability check + quota check + two directory records, atomic. **Metering is tenant-level but asynchronous**: space actors accumulate usage in space-local counters (puts, bytes, reads, lease-seconds) flushed with normal batches; a background rollup joins per-space usage through the directory into per-tenant aggregates on a coarse cadence — billing-grade, not correctness-grade; the verb write path never touches tenant state (per-prefix aggregates already supply per-space size signals for storage quotas). Leases live in slatedb like everything else: grant + counters commit in one atomic write batch, so epoch monotonicity and the lease table survive crashes exactly (recovery = WAL replay, nothing special-cased). Group commit is explicit: a verb's reply awaits durability, but the store holds a small flush window (~1ms) coalescing batches submitted by concurrent space actors into one WAL flush — slatedb's flush-interval knob, not kernel machinery; per-space ordering is untouched because each actor awaits its apply before dequeuing the next verb. Cost: acquire waits on a WAL flush — fine on NVMe, tens of ms on S3; revisit with async-durable lease writes + restart grace ≥ max TTL if acquire latency ever demands it.

**DST commitment (interim discipline, full rig later).** Full deterministic simulation testing (Turso/TigerBeetle-style: whole system as a pure function of a seed — virtual time, seeded fault injection at every boundary, thousands of replayable seeds in CI) arrives with milestone 2 and runs continuously after. Until then, *every* implementation choice must keep it cheap: verbs take explicit `now` (server clock in exactly one code path); all IO behind traits (`OrderedStore` async, immediate-resolve in-memory impl); actors are runtime-agnostic — plain async loops over mailboxes, never spawning internally, so production drives them with tokio and the sim drives identical code with a single-threaded seeded stepper; no wall clocks, no OS entropy, no `HashMap` iteration order influencing behavior (BTree structures in kernel state as insurance); the model-based proptest suites double as the sim's invariant oracles. Retrofitting any of this is the expensive part of DST — we pay as we go instead.

**Honest-but-dumb:** SIV per-component key pseudonymization, *path-chained*: component *i* is encrypted under the subtree key derived from the components before it (`k_child = HKDF(k_parent, component)`), so identical words at different paths pseudonymize differently — sibling equality, which server-side matching requires, is the only equality that leaks; prefix relations survive at component boundaries, and handing out one subtree's derived key is the selective-access primitive; XChaCha20-Poly1305 values with coordination tags bound in the AEAD associated data — client-known fields only (device, device_seq, epoch, ver, key_epoch — the last rides in the value envelope beside the nonce, permanently 0 in v1: a reserved rotation upgrade path); admission_seq is server-assigned post-encryption, so admission-order splice protection comes from anchors, not AD; client-computed per-key ver chains (server enforces monotonicity); subtree anchors at generation boundaries for fork detection (full fork-consistency explicitly out of scope). Server clock exists in one code path (lease deadlines); server trust in clients: zero.

**Key hierarchy (decided): name key ∥ space keys — never one HKDF root; all keys are random, passwords only ever wrap.** Each space carries exactly two immutable random keys, delivered to the client as a **SpaceEnvelope**: `Encrypted { name_key, space_keys: Map<Epoch, Key> }` — from which `space_id = HKDF(name_key)` is derived and *verified at open, never stored*, making the id a commitment that defeats server-side space substitution — or `Plaintext { space_id }` for unencrypted spaces, where the id is asserted and its integrity is the deliverer's problem. The **name key** (SIV + per-subtree derivation, `k_child = HKDF(k_parent, component)`) is deliberately independent of the space keys — deriving it was rejected: it would weld every member to epoch 0 forever and foreclose history-withheld joins — and effectively immutable: pseudonym stability is load-bearing (leases, shape cursors, and partition-scoped token claims all live in pseudonym space); re-pseudonymizing is formally "migrate to a new space" (a big hammer, never routine). **V1 has no cryptographic revocation:** the epoch map exists in the type and the AAD tag format (`key_epoch`, permanently 0) as a reserved upgrade path, but member removal is enforced entirely at the reachability layer — revoke tokens and claims, and still-valid keys decrypt nothing new. What rotation would add is protection against an ex-member colluding with the server for *future* ciphertext; that threat is deferred to a roadmap tier, and the full rotation design — fresh-random value keys per epoch, saga-ordered key delivery, park-and-refresh, epoch pointer, eager rewrite for mutable-SQL generations vs. lazy coexistence for append-only prefixes — is archived as designed-but-deferred, with MLS/TreeKEM noted for large-membership scale. **Namespace discipline (decided):** leading components are single-byte enum discriminants, never strings — top level `Namespace::{System = 0x00, User = 0x01}`, then `SystemScope::{Global, DeviceLocal}` and record kinds below System. App tuple `(a,b,c)` → `(User, a′,b′,c′)` (prepended by the codec); the link-space root envelope at `(System, Global, Envelope)` — the sole reserved plaintext key; per-device system records under `(System, DeviceLocal, device, …)` (device ids plaintext — the server already knows them from tags and leases). Disjoint top levels make user/system collision structurally impossible (no user input or pseudonym ever reaches the top level), cost one of the 16 components, sort machinery before data, and split the lease domains: the whole-db write lease is `(User,)`, so system-record writes never contend with data writers. System writes are ordinary writes — every put needs a covering write lease, no carve-outs; device-scoped prefixes (ledgers, per-device system records) hold long *stealable* write leases auto-acquired by their device, uncontended by construction and reclaimed across restarts via steal, which epoch-fences the device's own zombie incarnation. Recorded tension for the threat model, softened by path-chaining: whoever mints partition-scoped tokens must hold pseudonymization keys (claims are pseudonym-space prefixes), but only the *subtree keys* for the partitions it mints under — the derivation tree doubles as the least-privilege boundary for minters. Moot for whole-space claims; hosted auth needs only the subtree keys the tenant delegates, never the space's root name key.

## multilite — the SQL layer (where all the smarts live)

**Shipping name: multilite.** The SQLite-on-homebase product ships publicly as **multilite** (multi-writer + lite). During kernel-first development it will live in a sibling repo; this monorepo holds homebase only until multilite splits out. Public announcement scope, artifact gates, and the rusqlite/C-ABI claims are tracked in [LAUNCH_CHECKLIST.md](./LAUNCH_CHECKLIST.md) — every sentence in the launch post must have a reproducible artifact behind it.

**Single-file doctrine (decided July 2026): a multilite db is a regular SQLite file.** All client state — the one oplog, the one shared device_seq, per-space watermarks, held leases, codec caches, identity — lives in system tables inside the *same file* as the data it describes; "one fsync domain" becomes literal: a commit's user rows and its oplog entry are one SQLite transaction. Three consequences, embraced: **attach** — any existing SQLite file opens under multilite, which creates its system tables and prepares sync; **detach** — any multilite file opens under plain sqlite (system tables are ordinary tables, data intact; a multilite re-open after foreign writes is not guaranteed); **db-local by policy, not by schema** — the homebase client serves any number of spaces, with client-level singletons where one suffices: one device id, one shared `device_seq` stream, and one oplog whose commits each carry their space (the server requires only strictly-increasing-per-(space, device) with gaps legal, and every space sees a strictly increasing subsequence of the one counter and its commits in order from the one queue); multilite's policy is two spaces — the database plus its companion link space — enforced above the schema. **Vers (decided): assigned by the meta store — one Lamport high-water, no per-key table.** Per-key monotonicity needs no per-key state: commit stamps entries with consecutive vers above the high-water (in entry order, so duplicate keys in one batch behave like a sequence — the kernel's own within-batch rule), pulls raise the high-water to the maximum ver observed, and the acquire-barrier rule guarantees the counter dominates the stored ver of every key a writer may touch. User tables stay pristine — no extra or hidden columns, ever (attach, `SELECT *`, vanilla INSERTs, and detach all forbid it); sync metadata lives only in `hb_*` system tables. **Option A, noted for the multilite layer:** a per-row shadow tag table (`hb_rows(tbl, pk → device, seq, ver, admission, key_epoch)`) would buy full-strength per-row rollback detection against a lying server, provenance, and resumable incremental apply — additive to counter assignment, purely a multilite decision, and priced honestly: index entries are kernel keys too, so shadow tags would cover them as well (the dominant size cost; under the counter they are covered for free). Client durable truth is a **trait of transitions** (`MetaStore`: load once, then genesis/commit/trim/advance/lease-churn — each method one atomic, durable, async step; multi-transition flows follow the saga rule, idempotent and resumable). Commit self-assigns: the queue is keyed by the wire `DeviceSeq`, stamped at commit atomically with the entry — write-ahead by construction, no separate reservation — which rests on the contract that a store-backed client writes the server exclusively through its queue (storeless engine-tier consumers are separate devices): the storage representation is entirely the implementation's business. The reference implementation rides any `OrderedStore` (memory for tests, the sim's fault store for crash torture) under a private brand byte so cohabitants can share the store; multilite implements the trait natively as *legible SQLite system tables* (`hb_oplog(commit_seq, space, entries)`, `hb_leases(…)`, …), each transition running inside the transaction already writing the user's rows. One recomputation oracle (`certify`) and one conformance suite gate every implementation. **Device identity (leaning, ratify with the engine batch): a random id, minted unless supplied from outside, stored in the file** — stable across moves and renames, copied by file copies. Environment-derived ids (path/host hashes) were rejected: ambient, seed-untestable, and unbindable into future token claims. Fork safety is downstream, not preventive: with write-ahead seq reservation a legitimate resume never regresses, so an *unexpected `DeviceSeqRegression` is proof of a fork* — the engine re-mints the device id, resyncs, and requeues the oplog under the new identity (device ids are disposable; a fresh id starts at seq 1, old leases expire by TTL). Optional earlier warnings (inode/host heuristics, a per-device incarnation lease) are hardening layers, not requirements.

**State = shape cache ⊕ oplog.** Shapes: SQL-vocabulary subscriptions compiled to index-prefix ranges, each at its cursor; eviction is shape-granular. Oplog: unshipped committed ops, persisted inside the local SQLite file (one fsync domain), drained eagerly. Owned regions are authoritative; unowned are snapshots.

**Witness compiler (the novel contribution):** speculative execute in savepoint → concrete write set via preupdate hook (catches triggers/cascades) → derive leases (row write; uniqueness witness write on collated value; FK parent read) → sorted batch-acquire (no deadlocks) → barrier → re-execute → coverage check → retry budget → table escalation. Handles collations, expression/partial indexes, read-beyond-row triggers (static scan → escalate), rowids (per-device block allocation). Query-side twin compiles reads to range fetches (PK/index point-gets, FK closure; unindexed → subscribe-table fallback). Compat: swap the open call; same dialect (it *is* SQLite); `SQLITE_BUSY` for contention/offline; DDL requires db-level write lease.

**Partitions — the unifying quantum.** The partition is simultaneously the unit of access control, of developer-visible leasing, of shape subscription, and of key layout. A table declares its partition dimensions via a structured comment directive in its DDL (`-- @partition(tenant, org, band)`); the compiler prepends these components to the storage keys of every row and every index entry, so a partition is a literal key prefix spanning all physical representations of its rows. RBAC then reduces to prefix-scoped tokens: a principal either holds (read or write) claims on a partition prefix or does not — the partition is the quantum of authorization, and homebase enforces nothing finer. Sub-partition policy (field-level write rules, value-dependent visibility) is out of scope by construction and routes to schema refinement or a trusted validator tier. Partition dimensions must therefore be chosen by the policy surface, not data-modeling aesthetics: they should encode stable ownership boundaries (tenant, org, container, device), never user-rearrangeable structure (folders, status), because changing a partition column is a key move — delete and reinsert of the row and all its index entries. The directive lives in a comment rather than extended DDL grammar so that every `CREATE TABLE` remains byte-valid vanilla SQLite and the annotation replicates automatically with the schema via `sqlite_master`.

**Developer-visible leasing (the `LEASE` statement family).** Explicit leases are exposed at partition granularity only, through a small statement family that extends SQLite's grammar without altering any native statement's behavior (first-token dispatch on the reserved verb `LEASE`; everything else passes to SQLite unmodified): `LEASE READ|WRITE ON DB <name> [TABLE <t>] [PARTITION (dim = val, …)] [DURATION <interval>]`, plus `LEASE RELEASE` and `LEASE SHOW`. Acquisition is all-or-nothing with `SQLITE_BUSY`-style conflict reporting; renewal is engine-managed (batched heartbeats) rather than a per-statement knob, so `DURATION` means exactly "my disconnected write budget." This is the developer's entire lease vocabulary — pin a database for offline work, hold a device's ledger partition, observe who is editing what. All finer-grained coordination (sticky row W leases, uniqueness witnesses, FK parent R leases, range asserts) is compiler-internal machinery, invisible in the way a server engine's record locks are, and is structurally confined within the writer's claimed partitions: because indexes carry partition components, every internal lease the compiler takes falls inside a prefix the token already grants, and lock escalation is therefore capped at the principal's authorization — contention behavior can never exceed access. The one deliberate exception is a constraint declared globally unique across partitions, whose witness keys live at a reserved shared prefix that the token minter includes automatically for the table's writers.

**Consistency & durability knobs (decided).** Two pragmas harden the defaults when an app wants less trust in local state: (1) **remote-sync commit** — a client-library pragma (per connection or table) that withholds the local commit acknowledgment until the batch is remotely admitted, turning the 0-RTT locally-durable write into a 1-RTT remote-durable one; the async default and its watermark API remain for everything else. (2) **strong reads / authority re-verification** — a mode that forces reads through the consistent tier (a real `read_at` cut) and/or a policy that re-verifies authority on a short cadence (every few seconds: renew + epoch check confirming held leases are still live), so "owned, 0 RTT" reads carry a bounded window since the engine last proved it still holds what it thinks it holds. Both are per-app policy, not architecture: they trade latency for skepticism on top of the same verbs.

**Open and library shape (decided).** Multilite is drop-in for code, one decision for deployment: `multilite.open("hb://space/db?lifecycle=whatsapp&durability=local", key)` replaces the SQLite open call one-for-one, with all policy vocabulary — lifecycle preset, durability tier, heartbeat cadence — carried as connection-string options over working defaults (single-writer forever-W lease auto-acquired at open, engine-managed renewal), so a vanilla app runs unchanged and leases, partitions, and shapes are opt-in escalations rather than prerequisites. The irreducible deltas from vanilla SQLite are one extra argument (identity: a key/token must come from somewhere), one extended error meaning (`SQLITE_BUSY` can now mean offline or fenced), and the documented physics (durability watermark, freshness tiers). Structurally the library is Rust wrapping real libsqlite3 in-process — capture via preupdate/commit hooks and EQP, never a reimplementation — which yields three packaging tiers: a rusqlite-compatible Rust-native crate first; then `libmultilite` exposing the standard `sqlite3_*` C ABI (our open returns a genuine `sqlite3*` with machinery attached via hooks, so existing C-API consumers in Python/Ruby/PHP run unmodified except the open — the same distribution trick libSQL and SQLCipher use); then idiomatic per-language clients, Python first. One flagged risk for the C-ABI tier: the fixpoint must intercept COMMIT inside SQLite's commit-hook + savepoint choreography rather than at a wrapper boundary — feasible by design, but it constrains the shim and warrants an early spike.

**Known asymmetry:** leasing gets first-class statement syntax while partitioning rides a comment directive. This is deliberate but admittedly inelegant — `LEASE` is a novel statement with nothing to collide with, whereas any keyword inside `CREATE TABLE` would break vanilla-SQLite validity and sever the free schema propagation through `sqlite_master`. Revisit if the directive approach proves error-prone in practice.

**Conformance testing.** Multilite's dialect claim ("it *is* SQLite executing") is verified against SQLite's own public conformance corpus, **SQL Logic Test**, run through the Rust `sqllogictest` crate (implement one driver trait over a multilite connection; no C harness, no emulation) against a **vendored, pinned snapshot** of the corpus with a published **exclusion manifest** — the test families that fail by documented physics rather than by bug (journal/checkpoint pragmas we own, dense-rowid assertions vs. block allocation, multi-connection locking with extended BUSY meanings, schema dumps seeing `_hb_*` tables), each with its one-line reason. The corpus runs as a **conformance matrix** rather than a single pass: topology (local-genesis no-server / embedded kernel / HTTP server / HTTP with a second replica syncing concurrently) × encryption (off/on) × durability tier — with the invariant that every cell produces byte-identical results, plus a cross-cell check that a freshly bootstrapped replica re-runs the read-only queries identically (replication fidelity measured by SLT itself). File fidelity is CI-enforced in both directions: after each write workload, `detach` and reopen the file with stock libsqlite3 via rusqlite — asserting clean open, identical query results, passing `integrity_check`, and no additions beyond `_hb_*` tables — and, in reverse, a vanilla-produced file is *adopted* mid-suite (exercising the rowid-headroom scan) and the run continues. Two supporting jobs round it out: SQLite's own `testfixture` built from our exact amalgamation and compile flags (validating our vendored engine build, scoped to that), and our differential harness (same script minus `LEASE` statements against multilite vs. vanilla, byte-identical results) covering the DML/DDL replication paths SLT's read-heavy corpus doesn't reach.

**What's novel vs. not:** leased-KV is Chubby lineage. The deltas: (1) mandatory enforcement fused with storage + barrier (vs. advisory locks); (2) the witness compiler — externalizing key-range locking onto a lease service so unmodified SQLite keeps constraints multi-writer (cr-sqlite/Ditto abandon them for CRDTs; no prior art found); (3) coordination over ciphertext.

## What else the kernel serves (homebase as a primitive)

Each is a thin client library over the same seven verbs — no kernel changes:

- **Job queue (SQS-lite)** — write lease on the job key = claim; TTL = visibility timeout; completion put, then release; crash = expiry reclaim; epoch = zombie-worker fencing. *Adopted as rung 2.5 dogfood consumer: a few hundred lines, no SQLite, exercises TTL/contention/fencing under crash injection before the compiler exists.*
- **Distributed cron** — write lease on `("cron", job, slot)` + idempotent fired-marker = exactly-once firing across N schedulers.
- **Leader election / service registry** — long write lease = leadership; epoch = the fencing token handed downstream; the Chubby use cases, E2EE-compatible.
- **Kafka-lite** — admission seq = total order; cursors = consumer offsets; consumer groups = write-lease claims on partition prefixes. Encrypted event streaming at team scale.
- **Config / feature-flag distribution** — admin holds write lease, fleets poll `read_at`: consistent multi-key config cuts, never torn flag combinations; range seq = free config version.
- **Dropbox-lite** — content-addressed chunks at `("cas", hash)` + manifest keys; E2EE file sync. (The consumer that forces the deferred blob/value-size decision.)
- **CRDT sync backend** — per-device write prefixes carrying Automerge/Yjs ops; drop-in encrypted relay for the existing local-first ecosystem. Adoption wedge.
- **Secrets vault, IoT state, document store** — as previously mapped.

## Trust & platform: identity, links, bootstrap

**Doctrine.** The space is the primitive: an ownerless cryptographic object where key possession is readability and token claims are reachability — a space "belongs" to someone only the way a file belongs to a directory entry. Keys carry identity; ids derive (commitments, never mutable). App code never authenticates itself — roots are WebPKI + user password. Hosted auth (v1 cut: email/password with its own server-side verifier, **deliberately independent of the envelope salt below** — the envelope is reachable only after auth, so one derivation cannot serve both halves; OPAQUE later) mints per-tenant Ed25519 tokens with (space, prefix, device, exp); BYO backend = same claims, one verifier. Signup spam = friction economics (rate limits, verification-gated provisioning, quotas, App Attest knob); recovery = escrow-or-code tenant knob.

**Links and the directory.** Above spaces sits an optional convention layer: a **Link** — person, tenant, fleet, or agent pool — is a stable random Ed25519 keypair (the X25519 sealing key derives from it by the standard map, so one commitment covers both signing and sealing) with `LinkID = H(link_pub)` — the id is again a commitment, and doubles as the out-of-band safety number. Every Link owns a **Link space** at that id — itself a Plaintext-enveloped space, since each value is independently protected: the root envelope at `(System, Global, Envelope)` holds `{salt, link_pub, wrap: {method: Kek(…) | Hardware(…)}, wrapped link_priv}` (the method discriminant admits hardware keys and enclaves as protectors, never as protocol changes); the **directory** maps SIV-pseudonymized dbnames (under a key derived from link_priv) to entries sealed to link_pub, each carrying `{NameKey, SpaceKey}` — keys carry identity, ids derive; and an **inbox** prefix lets third parties, holding only link_pub and a sender-scoped claim, deposit invites the owner curates into named entries. **Link-space values are writer-signed (decided):** every value in a link space carries a signature verifiable against a *link* key, never a token identity — directory and curated entries signed by the owner's link_priv (checked against the link_pub the bootstrap already pinned via `H(link_pub) = LinkID`), inbox deposits signed by the sender's link key (checked against the sender's resolved-and-pinned link_pub, so curation is informed). This keeps the compromise asymmetry honest: an auth compromise can vandalize link-space records — visible denial, never silent key substitution — because valid tokens cannot forge link signatures. Signatures give authenticity, not freshness: rollback (a stale-but-validly-signed envelope or entry served to a joining device) is the residual gap, folded into the key-transparency roadmap item. Data spaces are exempt from per-writer signing: value AEAD already means only key-holders can mint valid ciphertext; writer signatures there are the operator-hostile tier's continuous-attestation extension, not v1.

**Bootstrap.** New-device bootstrap is one chain from (alias-or-LinkID, password): resolve → authenticate (the independent auth leg) → open Link space → fetch envelope → verify `H(link_pub) = LinkID` → `KEK = Argon2id(pw, salt)` → unwrap link_priv → read directory (verify signatures, decrypt names, unseal entries) → open children (verifying each `space_id = HKDF(name_key)`). Alias resolution is the **Resolver** — identity, not auth: `(namespace, alias) → {link_id, link_pub, bound_at, proof}` — with self-signed binding proofs, mandatory client-side TOFU pinning (hard-warn on key change), and key transparency as the systemic roadmap fix; the protocol and a reference resolver (alias rows stored through the kernel itself) are OSS, the hosted resolver with verified alias claiming is commercial trust infrastructure. The trust architecture reads as an assembly line — identity mints bindings → auth mints tokens → reachability gates bytes — with readability (keys) running parallel, touching identity at exactly one object, link_pub; the compromise asymmetry is the threat model's spine: auth compromise leaks reachability to ciphertext; resolver compromise redirects keys (TOFU pins bound the blast radius to first contact).

**Lifecycle.** Unlinked creation is local genesis: mint keys, derive id, implicit forever-W, keys in the platform keystore plus local registry — with the stated physics that device death is total loss, which is precisely what linking fixes. Linking is a persisted, idempotent, client-side saga (no cross-space atomicity exists, and a global oplog was rejected for smuggling it back in), ordered by the normative **keys-before-data** rule: directory entry first, then registration, then snapshot upload, then the gen-0 manifest — so sealed keys reach durable server storage before the first data byte, making the worst crash a visible dangling entry rather than permanently unreadable ciphertext. IDs never mutate; attach-without-link is permitted but flagged as keys-not-backed-up.

**Code layout.** The identity layer folds into existing crates as modules, not new crates: `homebase-client` gains `homebase_client::identity` with the enforced module rule that core client modules never import it; `homebase-core` takes the binding/proof types; `homebase-wire` carries the protocol types; the server crate (`homebase`) gains a resolver module beside the kernel — the client/server crate split then enforces the security invariant that the resolver can never link custody code, and the `homebase` binary composes kernel + resolver as subcommands (supersedes the earlier planned `homebased` binary). `multilite` is the SQL crate in the same monorepo. Promotion of modules to crates waits on a tripwire: an external consumer needing identity without the client.

## Naming, packaging, and the OSS boundary (decided July 2026)

**Naming, repo, and crate layout.** The product brand is **multilite** ("SQLite with end-to-end encrypted sync"), and it owns the launch surface: the monorepo lives at `github.com/multilite/multilite` so that install-name, star button, and announcement all concentrate on one word — "multi" reading honestly as *multi-device* at launch and graduating to *multi-writer* when the constraint gauntlet lands. Inside it, **homebase** is the coordination kernel, laid out server-eponymously: `homebase` is the **server crate** — space actors, augmented tree, `ShardLog` impls (local-NVMe WAL and S3/slatedb), resolver as a sibling module — shipping as lib plus the **`homebase` binary**, subcommand-structured (`homebase serve`, `homebase resolver`), so `cargo install homebase` yields the thing you deploy; `homebase-client` is the SDK — spaces, `SpaceEnvelope`, leases, sync, with the identity layer folded in as `homebase_client::identity` under the enforced module rule that core client modules never import it, and an optional `embedded` feature depending on the server crate (as lib) for in-process kernels; `homebase-core` carries shared types, tuple keys, and binding/proof verification; `homebase-wire` the protocol types and transport scaffolding; `multilite` the SQL layer atop `homebase-client`. Kernel extraction to its own repo is deferred behind a tripwire — the first non-SQL consumer (the agent substrate) — at which point `git filter-repo` carries the kernel crates, test corpus, and `specs/` out as a standalone project. **homestead** is reserved as the eventual company name; commercial offerings always lead with that cleared brand, never "Homebase <product>."

**OSS boundary, topologies, and wire protocol.** The license line follows the single-operator/multi-principal doctrine: **Apache-2.0** covers everything a single operator needs — kernel, both storage tiers with manifest-CAS failover, client, multilite, embedded mode, the E2EE codec, and crucially the full **scope-enforcement machinery** (claims format and prefix checks stay in-kernel, proptest-covered); the OSS `homebase` binary is deliberately single-tenant, single-shard, and *verification-free* — it trusts its ingress (localhost/VPC/mTLS, optional static secret) with callers asserting `device_id`. **Proprietary** (closed until a partner forces BUSL) is the multi-principal layer: token minting and verification, identity/SSO, hosted resolver, and the fleet/operator plane (shard assignment, migration, hot-standby orchestration, metering-to-billing) — code a managed-service cloner needs and a self-hoster never runs. Two blessed OSS topologies share one doctrine, *the app's server sits on the authorization path, never the data path*: **(1) proxied binary** — the app's web server fronts `homebase`, performs its own access control, and injects per-request scope assertions (the same claims shape the commercial front door fills from real signed tokens; a ~200-line `homebase-scope` axum middleware example ships in-repo); **(2) S3-direct** — the current writer runs the embedded kernel as S3's sole author, readers pull manifest + SSTs directly using **STS temporary credentials vended by the app's server**, takeover via manifest CAS; scoping here is space-granular (bucket prefixes), with sub-space RBAC requiring topology 1. The proxy topology decided the transport: **HTTP, not gRPC** — plain HTTP/1.1+2 that stock middleware can route, inspect, scope-check, and rate-limit: JSON for low-rate control verbs (acquire/renew/release), compact binary bodies (bincode/CBOR) for `put_batch`/`sync`/`read_at`, SSE or long-poll for the sync stream (contention signals still ride renewals; no push channel). This adds one normative wire-spec section: a canonical, middleware-legible encoding of key prefixes in requests, since proxies now match scope assertions against request keys. `homebase-client` abstracts it all behind one verb trait with three transports — HTTP, in-process embedded, S3-read-only — plus a `CredSource` trait (mirroring `KeySource`) for pluggable STS refresh.

## Milestones — kernel first, by design

The server is no longer trivially dumb (admission serialization, augmented range-max tree, atomic read_at cuts, lease overlap checks, fencing) — which *strengthens* kernel-first sequencing: it's now genuinely novel infrastructure deserving its own validation, and it's pure distributed-systems Rust with zero SQLite dependency.

1. **Kernel, in-memory** — client + server crates, all seven verbs, tuple encoding, augmented tree. *Exit: proptest invariant suite (overlap/mode compatibility, fencing, barrier, expiry asymmetry, ver monotonicity, read_at cut correctness). The crown jewel.*
2. **Kernel torture** — deterministic sim: virtual time, partitions, steal races, zombie writers, contended handoff. *Exit: 10k seeds clean.*
2.5. **Job-queue dogfood** — first real consumer, kernel-only. *Exit: crash-injected workers, zero double-completions, zero lost jobs.*
3. **Persistence + crypto codec** — storage trait (in-memory for the sim; slatedb behind it: local-NVMe object store dev, S3 prod); shard WAL group commit; one atomic write batch per verb (leases + counters + data uniformly); SIV/AEAD/HKDF codec. *Exit: kill -9 mid-batch → invariants hold; server dump = ciphertext only.*
4. **Single-writer SQL, logical** — one write lease on db prefix; hook capture; custom changesets; dumb-applier replicas; DDL-as-text; generation snapshots. *Exit: unmodified rusqlite app runs; checksum-matched replicas over randomized DML/DDL; ~1 RTT handoff.* (4.5 optional: hand-wired witnesses on a toy schema — falsify the mechanism cheaply.)
5. **Witness compiler + fixpoint + shapes.** *Exit: constraint gauntlet — concurrent colliding writers, zero silent violations.*
6. **Proof point** — 3 nodes, one DB, real Axum app; latency/throughput vs. Postgres; E2EE demo. Write-up leads with the compiler (mini-paper shape).

## Client build sequence (decided July 2026): interlocking contracts, deep implementation

Rebuilt from the kernel commit point in small atomic batches — each lands a crisp contract *plus* its full implementation and torture coverage; nothing merges half-tested. Three contracts interlock everything: **ServerHandle** (the verbs, space-qualified), the **client meta schema** (durable truth over `OrderedStore`), and the **codec** (privacy at ingest). Sequencing principles: contracts before consumers; the first real consumer (the job queue) validates coordination *before* crypto complexity enters; the codec lands before the replica write path because commit encrypts at ingest (plaintext-replica-then-retrofit was rejected — it reworks the oplog format mid-history); the engine is write-through against the schema from day one (single owner of durable truth; retrofitting persistence produces seq/lease dual-ownership); transport comes strictly after semantics, gated on an in-process ≡ remote conformance suite; identity composes everything beneath it and lands last, with alias resolution stubbed (LinkID-direct) until the resolver batch. The pre-rebuild working tree is a quarry, not a landfill: batches 1, 4, and 14 re-land already-tortured code nearly verbatim, and the interface sketches become each batch's contract.

1. **Storage vocabulary → core** — `OrderedStore`/`WriteBatch`/`MemoryStore`/conformance move to `homebase-core`; server re-exports. *Exit: workspace green, zero behavior change.*
2. **ServerHandle contract** — space-qualified verbs; closure blanket impl (in-process = `|id| server.space(id)`); uninhabited `Offline`. *Exit: verb conformance over an in-process server.*
3. **Client meta schema** — record layout (device id; per-space seq, oplog, watermark, leases, codec cache) + atomic-batch composers over the shared store. *Exit: recomputation-audit oracle — the client twin of the server's `check`.*
4. **Engine, write-through** — device_seq/replay fence, HeldLease local deadlines, subset renew, blind-retry rules — durable truth in the schema from day one, in-memory view only. *Exit: two-clock expiry, exactly-once replay-fence, and crashy-resume tortures with fault injection on both stores.*
5. **Job-queue dogfood (rung 2.5)** — first consumer, engine tier, no crypto. *Exit: crash-injected workers, zero double-completions, zero lost jobs; verdict on the storeless tier and `put_once`.*
6. **Codec** — path-chained SIV names, XChaCha value envelopes (`key_epoch` ≡ 0, tags in AAD), namespace enums, `SpaceEnvelope` + `space_id = HKDF(name_key)` verify-at-open. Pure. *Exit: roundtrip/tamper/wrong-key gauntlet; prefix-correspondence property.*
7. **Replica write path** — `commit` (encode-at-ingest + oplog append, one atomic batch), `push` (drain via engine), `discard`. *Exit: exactly-once through the codec under crash + ack-drop torture; ciphertext-only-at-rest oracle.*
8. **Replica read path** — `pull` (one consistent cut → decode → `Cut`), watermark advance, introspection. *Exit: equivalence torture — pull ⊕ replay(oplog) ≡ authoritative plaintext state against the server audit.*
9. **Client facade** — `open`/`open_offline` (store-owned device identity), `space`/`create_space` (local genesis), `replicas`. *Exit: multi-space isolation; offline-replica physics.*
10. **DiskStore** — redb-backed, directory-tied `OrderedStore`. *Exit: the shared storage conformance suite + kill-9 durability test.* (Slot-flexible: anywhere after 3.)
11. **Identity primitives** — Ed25519 link keys (X25519 sealing key derived), `LinkID` commitment, Argon2id KEK wrap with method discriminant, sign/seal. Pure. *Exit: test vectors + property tests.*
12. **Link-space formats** — root envelope record, directory codec (SIV names under a link_priv-derived key; entries sealed to link_pub *and* owner-signed), sender-signed inbox invites. *Exit: forged/tampered/replayed-entry rejection gauntlet — the auth-compromise key-substitution attack must fail in tests.*
13. **Linking saga + bootstrap** — local genesis → link (keys-before-data, idempotent, resumable), new-device chain (LinkID-direct), attach-without-link. *Exit: saga crash-torture at every step boundary; dangling-entry-never-unreadable-ciphertext oracle.*
14. **Wire re-land** — kernel proto + DTO conversions + the error-plane rule. *Exit: encoded roundtrip suite.* (Slot-flexible: re-lands as built.)
15. **Transport adapters** — tonic service over the kernel, `RemoteServer` client, UDS + TCP. *Exit: engine/replica suites re-run over loopback gRPC, observationally identical to in-process.*
16. **Resolver (reference)** — binding/proof types in core, resolver proto, server resolver module (alias rows stored through the kernel), client TOFU pinning. *Exit: bootstrap's resolve leg live; pin-change hard-warn tests.*

## Deliberate exclusions

No read→write lease upgrade, ever (release and re-acquire). No max_puts one-shot leases (dropped; claims are write-lease + marker patterns — revisit if the job-queue dogfood earns it back). No separate weak `sync` verb (read_at subsumes it: one admission sequence makes the consistent cut ~free; revisit only if incremental catch-up or long-poll delta feeds demand it). No server-side search/aggregation (permanent). No cross-lease-domain transactions. No optimistic CAS / range-CAS (range-version machinery exists via admission seqs; conditional writes noted as the future mechanism if an optimistic profile ever earns in — not built). No MVCC/server history (consistent reads are single-shot; pinned read transactions would need a retention window — deferred). No merge engine/CRDTs. No push channel (renewal piggyback suffices). Fork-consistency out of scope (anchors bound the window).

## Open decisions

**Wire protocol — superseded (July 2026): HTTP, not gRPC.** The earlier gRPC/tonic decision is reversed by the proxied-binary topology (see "OSS boundary, topologies, and wire protocol"): plain HTTP/1.1+2 that stock middleware can route, inspect, scope-check, and rate-limit. What carries forward unchanged from the gRPC design: the **error-plane rule** (kernel rejections travel in response payloads; transport statuses are reserved for the transport plane — the `SpaceError::Kernel` vs `Unavailable` split the client retry contract is built on), one client verb trait over multiple endpoint families (now in-process, HTTP, S3-read-only), durations-cross-as-ms, and timestamps-never-cross (asymmetric expiry). The tonic protos and DTO conversions built pre-rebuild remain in the quarry as reference; the wire batch re-lands as the HTTP spec instead, including the normative middleware-legible key-prefix encoding.

**TODO — resharding mechanism (deferred; settle before wire formats freeze).** Goal: start with few shards, add more under load, no stop-the-world rehashing. Two candidate architectures, decision deferred:
(1) **Generational addressing** — shard id is the triple `(gen, num_shards, shard_idx)`: `gen` a monotonic reshard generation, `num_shards` that generation's count (power of two, so placement is `hash(space) mod num_shards` and gen-to-gen ownership is a bit-prefix refinement), `shard_idx` the slot. Scaling = bump the generation, stack the new gen's SST/WAL lineage *in front*; reads consult newer gens first and fall back through older ones until offline compaction migrates data forward and retires them — LSM semantics lifted to fleet level. Costs: read amplification across live generations; a space can't actually straddle generations (counters/leases/aggregates need read-your-writes), so each space needs a whole-space cutover anyway.
(2) **Directory with default placement** — exploit that the space is the natural movement quantum: no verb spans spaces, and the space id is the first key component, so a space is one contiguous byte range in shard storage. Placement = small directory (space → shard) holding only exceptions, default `hash(space) mod num_shards` at the current generation; growing = bump the default and migrate spaces wholesale (lazily on first touch or by background sweep); hot spaces movable individually by directory entry.
Leaning (2) — it drops the multi-generation read path and adds per-space placement freedom — but either way the shard-id *type* should carry the triple from day one so wire formats and manifests never migrate. Fits post-v1 next to space→shard placement/rebalancing (slatedb manifest views may give O(1) key-range splits for the copy step).
