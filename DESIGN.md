## What it is

**An opaque, end-to-end encrypted operation log for building local-first databases, with optional prefix leases for race prevention rather than mandatory write capabilities.**

The current Homebase client is an operation-log service, not a managed KV replica: applications submit local mutation batches, push them asynchronously, pull the server's exact admission log, and decide when their own durable state has applied each admitted batch. Leases reserve prefix ranges against conflicting work and substantiate checked range assertions, but an unreserved write is admissible whenever it conflicts with no live foreign lease. The future multilite layer will own SQLite capture and application above this kernel.

**Why E2EE is a moat, not a feature:** server-side query execution requires server-side plaintext (uniqueness needs equality, WHERE needs reading, indexes need order). Therefore E2EE ⇒ client-side execution ⇒ client-side enforcement ⇒ something lease-shaped. Authority-model systems cannot follow even in principle.

**Honest scope:** Homebase currently guarantees operation admission and replay, not SQL correctness or automatic replica convergence. Applications may queue submissions offline, but checked invariant-dependent writes need a live reservation and an applied lease barrier before local submission; remote admission remains the final authority. Hot prefixes are a poor fit for coarse long leases. Server-log retention is unbounded until an authenticated checkpoint/GC protocol exists.

**Kernel guarantees:** each admission request is atomic; device sequences and cumulative checksums detect replay, gaps, and divergent device history; per-key/range versions reject regressions; range assertions reject unseen foreign history; and `pull` returns a dense exact prefix of server history. These are ingredients for a SQL layer, not an ACID claim by themselves.

## homebase — the kernel

**The kernel in one line: an opaque materialized KV view plus an exact append-only admission log.** One ordered map per space holds tuple keys (`Vec<Bytes>`, component-wise prefixes, order-preserving flat encoding) and opaque values. Every Set, Delete, and DeleteRange carries a client-assigned `DeviceTag`, detached `Seal`, and server-assigned `AdmissionTag`; the server also maintains range tombstones, lease indexes, per-device sequence/checksum state, and prefix aggregates for history, version floors, and live counts. The transport-neutral verbs are:

```
acquire/renew/release/list_leases  # synchronous lease reservation and repair
admit                              # atomic data batches; versions, assertions,
                                   # reservation conflicts, and checksum enforced
pull                               # dense exact full-space admission history
get / list
read_at(ranges, cursors) -> (S, cuts) # stateless arbitrary-range observation
```

Prefix-scoped authorization is a transport concern; leases are coordination state. **Two lease modes:** write conflicts with every overlapping lease, while read coexists with read and conflicts with write. There is no stealing, fencing epoch, or pre-deadline takeover. A write needs no lease of its own, but admission rejects it when a live foreign lease reserves an overlapping target; range mutations check overlap in both ancestor and descendant directions. A checked range assertion additionally requires the client to hold a live covering lease, to have applied the admit log through that lease's barrier (`neck > barrier`), and to assert an inclusive foreign-history bound `upto` no newer than its applied coverage. Acquire, renew, release, and repair remain synchronous control operations outside both data logs. Local lease authority deliberately expires early by `max(TTL / 1000, 10ms)` so client authority remains a subset of the server's.

**Deployment shape:** a server owns **shards** of spaces; a shard is one ordered store, group-committing across its spaces (storage key = space id ⊕ record kind ⊕ encoded key). **Each space is an actor**: one mailbox, one logical thread; verbs for a space are dequeued and run strictly one at a time, with `now` stamped at dequeue. Admission sequences, lease transitions, materialization, exact-log append, aggregate updates, and per-device checksum advancement therefore share one linearization order. Actors are runtime machinery, not identity: all correctness state remains in the ordered store.

**Tenancy & space sizing.** A space is the unit of every guarantee: one admission sequence, lease table, exact log, materialized view, and actor. Each `(space, device)` has an independent submit sequence and `DeviceChecksum`; each client keeps an independent admit-log cursor triple in the server's `AdmissionSeq` domain. Nothing is atomic across spaces. Above spaces, a future tenant/control plane may own keys, quotas, billing, and shard placement; the kernel itself remains tenant-oblivious.

**Directory & metering (tentative — deferred until the commercial tier; finalize then).** Multi-tenancy, the directory, and metering are not needed for the OSS/single-operator milestones. A future directory can map spaces to tenants and shards without changing kernel request bodies. Lease grants, renewals, releases, data admission, exact-log append, materialization, aggregates, and counters remain ordinary atomic ordered-store batches; no correctness state is special-cased in memory. Group commit and object-store persistence are deployment optimizations that must preserve each actor's per-space order.

**DST commitment (interim discipline, full rig later).** Full deterministic simulation testing (Turso/TigerBeetle-style: whole system as a pure function of a seed — virtual time, seeded fault injection at every boundary, thousands of replayable seeds in CI) arrives with milestone 2 and runs continuously after. Until then, *every* implementation choice must keep it cheap: verbs take explicit `now` (server clock in exactly one code path); all IO behind traits (`OrderedStore` async, immediate-resolve in-memory impl); actors are runtime-agnostic — plain async loops over mailboxes, never spawning internally, so production drives them with tokio and the sim drives identical code with a single-threaded seeded stepper; no wall clocks, no OS entropy, no `HashMap` iteration order influencing behavior (BTree structures in kernel state as insurance); the model-based proptest suites double as the sim's invariant oracles. Retrofitting any of this is the expensive part of DST — we pay as we go instead.

**Honest-but-dumb:** path-chained deterministic component pseudonymization preserves only the prefix/equality structure the server needs. XChaCha20-Poly1305 values use detached `Seal` metadata (`scheme`, nonce, authentication tag, and reserved payload), with scheme, explicit operation kind, anonymized target, device, device sequence, version, and `cipher_epoch` bound in AEAD associated data. Set encrypts a non-empty versioned frame so an empty logical value still has ciphertext; Delete and DeleteRange authenticate empty plaintext and carry no ciphertext. `admission_seq` is server-assigned after sealing. The per-device cumulative checksum detects lost acknowledgements, honest-server rollback, and duplicate-device history divergence; completeness against a Byzantine server requires a future external witness or authenticated whole-space checkpoint.

**Key hierarchy (decided): name key ∥ space keys — never one HKDF root; all keys are random, passwords only ever wrap.** An encrypted `SpaceEnvelope` carries a name key and a map of value keys by `CipherEpoch`; `space_id` is derived from the name key and verified at open. A plaintext envelope carries only its asserted `space_id`. The name key is independent and effectively immutable because pseudonym stability is load-bearing for server keys, leases, range cursors, and prefix authorization. **V1 has no cryptographic revocation:** `cipher_epoch` is permanently 0 but remains in `DeviceTag` and AEAD context as an upgrade path. Rotation, history-withheld joins, re-pseudonymization, identity distribution, and namespace conventions belong to later layers. Leases reserve prefixes against conflicting writers; they are not capabilities required for every write, and no pre-deadline stealing path exists.

## multilite — the SQL layer (where all the smarts live)

**Shipping name: multilite.** The SQLite-on-homebase product ships publicly as **multilite** (multi-writer + lite). Its SQL layer and language skins live beside Homebase in this monorepo. Public announcement scope, artifact gates, and the rusqlite/C-ABI claims are tracked in [LAUNCH_CHECKLIST.md](./LAUNCH_CHECKLIST.md) — every sentence in the launch post must have a reproducible artifact behind it.

**Single-file doctrine (multilite target): a multilite database is a regular SQLite file.** Client identity, per-space submit/admit logs and cursors, version high-water, held leases, and codec cache live in system tables in the same file as application data. A local SQLite transaction can therefore persist user rows and its Homebase submission in one fsync domain; incoming application changes and admit `neck` can likewise advance atomically. Homebase already expresses this durable vocabulary through `MetaStore`, while the current reference implementation stores it over any `OrderedStore`. Each space has independent `DeviceSeq`, `AdmissionSeq`, `DeviceChecksum`, and version domains. An unexpected sequence/checksum divergence is a fatal fork reported to the caller; Homebase does not silently remint identity or requeue ambiguous history.

**State = application state ⊕ submit log ⊕ admit log.** Homebase persists outgoing `DeviceOp` batches and incoming exact `AdmittedBatch` history but does not apply either to application tables. The future multilite layer must atomically pair each local SQLite commit with its submit entry, and must durably apply each incoming batch before advancing admit `neck`. Arbitrary range fetch cursors remain application-owned and never establish replication progress.

**Rebase analysis is assertion analysis.** `Space::analyze_rebase(from..to)` compares the active submit suffix with exactly one caller-selected retained admit interval, without network I/O or cursor movement. Existing `RangeAssert`s are the only declared local dependencies: newer foreign point mutations below an asserted prefix and overlapping DeleteRanges produce per-`DeviceSeq` failures; unasserted writes do not conflict by inference. The result carries the analyzed interval and observed submit/admit cursors for higher-layer freshness checks. Homebase gives neither the interval nor admit `neck` implicit reconciliation meaning: the application owns interval continuity and application policy. It also assumes each assertion was truthful against application state outside the interval, because Homebase neither owns that state nor reconstructs undeclared dependencies from mutation keys.

**Witness compiler (the novel contribution):** speculative execute in savepoint → concrete write set via preupdate hook (catches triggers/cascades) → derive leases (row write; uniqueness witness write on collated value; FK parent read) → sorted batch-acquire (no deadlocks) → barrier → re-execute → coverage check → retry budget → table escalation. Handles collations, expression/partial indexes, read-beyond-row triggers (static scan → escalate), rowids (per-device block allocation). Query-side twin compiles reads to range fetches (PK/index point-gets, FK closure; unindexed → subscribe-table fallback). Compat: swap the open call; same dialect (it *is* SQLite); `SQLITE_BUSY` for contention/offline; DDL requires db-level write lease.

**Partitions — the unifying quantum.** The partition is simultaneously the unit of access control, of developer-visible leasing, of shape subscription, and of key layout. A table declares its partition dimensions via a structured comment directive in its DDL (`-- @partition(tenant, org, band)`); the compiler prepends these components to the storage keys of every row and every index entry, so a partition is a literal key prefix spanning all physical representations of its rows. RBAC then reduces to prefix-scoped tokens: a principal either holds (read or write) claims on a partition prefix or does not — the partition is the quantum of authorization, and homebase enforces nothing finer. Sub-partition policy (field-level write rules, value-dependent visibility) is out of scope by construction and routes to schema refinement or a trusted validator tier. Partition dimensions must therefore be chosen by the policy surface, not data-modeling aesthetics: they should encode stable ownership boundaries (tenant, org, container, device), never user-rearrangeable structure (folders, status), because changing a partition column is a key move — delete and reinsert of the row and all its index entries. The directive lives in a comment rather than extended DDL grammar so that every `CREATE TABLE` remains byte-valid vanilla SQLite and the annotation replicates automatically with the schema via `sqlite_master`.

**Developer-visible leasing (the `LEASE` statement family).** Explicit leases are exposed at partition granularity only, through a small statement family that extends SQLite's grammar without altering any native statement's behavior (first-token dispatch on the reserved verb `LEASE`; everything else passes to SQLite unmodified): `LEASE READ|WRITE ON DB <name> [TABLE <t>] [PARTITION (dim = val, …)] [DURATION <interval>]`, plus `LEASE RELEASE` and `LEASE SHOW`. Acquisition is all-or-nothing with `SQLITE_BUSY`-style conflict reporting; renewal is engine-managed (batched heartbeats) rather than a per-statement knob, so `DURATION` means exactly "my disconnected write budget." This is the developer's entire lease vocabulary — pin a database for offline work, hold a device's ledger partition, observe who is editing what. All finer-grained coordination (sticky row W leases, uniqueness witnesses, FK parent R leases, range asserts) is compiler-internal machinery, invisible in the way a server engine's record locks are, and is structurally confined within the writer's claimed partitions: because indexes carry partition components, every internal lease the compiler takes falls inside a prefix the token already grants, and lock escalation is therefore capped at the principal's authorization — contention behavior can never exceed access. The one deliberate exception is a constraint declared globally unique across partitions, whose witness keys live at a reserved shared prefix that the token minter includes automatically for the table's writers.

**Consistency & durability knobs (future multilite policy).** Homebase exposes mechanism rather than a special synchronous commit mode: submit locally, then call `Submission::push()` when that exact write must be remotely attributed before returning. Applications that consume incoming history pull it, durably apply each dense batch, then advance admit `neck`; stateless `fetch` never substitutes for that progress. Stronger SQL read policies must be defined by multilite in terms of its applied admit cursor and checked range assertions, not by treating lease possession alone as proof that application state is current.

**Open and library shape (future multilite).** The intended SQL layer wraps real libsqlite3 in-process, captures changes through SQLite hooks, and implements `MetaStore` in ordinary system tables. Connection options may select durability and lease policy, but acquiring a forever write lease at open is not a kernel default and lease loss is contention/availability, not epoch fencing. Packaging targets remain a Rust-native wrapper first and a C-ABI shim only after the transaction/apply choreography is proven.

**Known asymmetry:** leasing gets first-class statement syntax while partitioning rides a comment directive. This is deliberate but admittedly inelegant — `LEASE` is a novel statement with nothing to collide with, whereas any keyword inside `CREATE TABLE` would break vanilla-SQLite validity and sever the free schema propagation through `sqlite_master`. Revisit if the directive approach proves error-prone in practice.

**Conformance testing.** Multilite's dialect claim ("it *is* SQLite executing") is verified against SQLite's own public conformance corpus, **SQL Logic Test**, run through the Rust `sqllogictest` crate (implement one driver trait over a multilite connection; no C harness, no emulation) against a **vendored, pinned snapshot** of the corpus with a published **exclusion manifest** — the test families that fail by documented physics rather than by bug (journal/checkpoint pragmas we own, dense-rowid assertions vs. block allocation, multi-connection locking with extended BUSY meanings, schema dumps seeing `_hb_*` tables), each with its one-line reason. The corpus runs as a **conformance matrix** rather than a single pass: topology (local-genesis no-server / embedded kernel / HTTP server / HTTP with a second replica syncing concurrently) × encryption (off/on) × durability tier — with the invariant that every cell produces byte-identical results, plus a cross-cell check that a freshly bootstrapped replica re-runs the read-only queries identically (replication fidelity measured by SLT itself). File fidelity is CI-enforced in both directions: after each write workload, `detach` and reopen the file with stock libsqlite3 via rusqlite — asserting clean open, identical query results, passing `integrity_check`, and no additions beyond `_hb_*` tables — and, in reverse, a vanilla-produced file is *adopted* mid-suite (exercising the rowid-headroom scan) and the run continues. Two supporting jobs round it out: SQLite's own `testfixture` built from our exact amalgamation and compile flags (validating our vendored engine build, scoped to that), and our differential harness (same script minus `LEASE` statements against multilite vs. vanilla, byte-identical results) covering the DML/DDL replication paths SLT's read-heavy corpus doesn't reach.

**What's novel vs. not:** leased-KV is Chubby lineage. The deltas: (1) mandatory enforcement fused with storage + barrier (vs. advisory locks); (2) the witness compiler — externalizing key-range locking onto a lease service so unmodified SQLite keeps constraints multi-writer (cr-sqlite/Ditto abandon them for CRDTs; no prior art found); (3) coordination over ciphertext.

## What else the kernel serves (homebase as a primitive)

Each is a thin client library over the same seven verbs — no kernel changes:

- **Job queue (SQS-lite)** — write lease on the job key = claim; TTL = visibility timeout; completion submission, then unlease; crash = expiry reclaim. Homebase rejects a stale worker's later Homebase write while another device holds the reservation, but does not fence unrelated external side effects.
- **Distributed cron** — write lease on `("cron", job, slot)` + idempotent fired-marker = exactly-once firing across N schedulers.
- **Leader election / service registry** — a long write lease can coordinate leadership only for Homebase-mediated state. Downstream fencing tokens are a separate feature and are not supplied by lease ids or epochs.
- **Kafka-lite** — admission seq = total order; cursors = consumer offsets; consumer groups = write-lease claims on partition prefixes. Encrypted event streaming at team scale.
- **Config / feature-flag distribution** — admin holds write lease, fleets poll `read_at`: consistent multi-key config cuts, never torn flag combinations; range seq = free config version.
- **Dropbox-lite** — content-addressed chunks at `("cas", hash)` + manifest keys; E2EE file sync. (The consumer that forces the deferred blob/value-size decision.)
- **CRDT sync backend** — per-device write prefixes carrying Automerge/Yjs ops; drop-in encrypted relay for the existing local-first ecosystem. Adoption wedge.
- **Secrets vault, IoT state, document store** — as previously mapped.

## Trust & platform: identity, links, bootstrap

**Doctrine.** The space is the primitive: an ownerless cryptographic object where key possession is readability and token claims are reachability — a space "belongs" to someone only the way a file belongs to a directory entry. Keys carry identity; ids derive (commitments, never mutable). App code never authenticates itself — roots are WebPKI + user password. Hosted auth (v1 cut: email/password with its own server-side verifier, **deliberately independent of the envelope salt below** — the envelope is reachable only after auth, so one derivation cannot serve both halves; OPAQUE later) mints per-tenant Ed25519 tokens with (space, prefix, device, exp); BYO backend = same claims, one verifier. Signup spam = friction economics (rate limits, verification-gated provisioning, quotas, App Attest knob); recovery = escrow-or-code tenant knob.

**Links and the directory.** Above spaces sits an optional convention layer: a **Link** — person, tenant, fleet, or agent pool — is a stable random Ed25519 keypair (the X25519 sealing key derives from it by the standard map, so one commitment covers both signing and sealing) with `LinkID = H(link_pub)` — the id is again a commitment, and doubles as the out-of-band safety number. Every Link owns a **Link space** at that id — itself a Plaintext-enveloped space, since each value is independently protected: the root envelope at `(System, Global, Envelope)` holds `{salt, link_pub, wrap: {method: Kek(…) | Hardware(…)}, wrapped link_priv}` (the method discriminant admits hardware keys and enclaves as protectors, never as protocol changes); the **directory** maps SIV-pseudonymized dbnames (under a key derived from link_priv) to entries sealed to link_pub, each carrying `{NameKey, SpaceKey}` — keys carry identity, ids derive; and an **inbox** prefix lets third parties, holding only link_pub and a sender-scoped claim, deposit invites the owner curates into named entries. **Link-space values are writer-signed (decided):** every value in a link space carries a signature verifiable against a *link* key, never a token identity — directory and curated entries signed by the owner's link_priv (checked against the link_pub the bootstrap already pinned via `H(link_pub) = LinkID`), inbox deposits signed by the sender's link key (checked against the sender's resolved-and-pinned link_pub, so curation is informed). This keeps the compromise asymmetry honest: an auth compromise can vandalize link-space records — visible denial, never silent key substitution — because valid tokens cannot forge link signatures. Signatures give authenticity, not freshness: rollback (a stale-but-validly-signed envelope or entry served to a joining device) is the residual gap, folded into the key-transparency roadmap item. Data spaces are exempt from per-writer signing: value AEAD already means only key-holders can mint valid ciphertext; writer signatures there are the operator-hostile tier's continuous-attestation extension, not v1.

**Bootstrap.** New-device bootstrap is one chain from (alias-or-LinkID, password): resolve → authenticate (the independent auth leg) → open Link space → fetch envelope → verify `H(link_pub) = LinkID` → `KEK = Argon2id(pw, salt)` → unwrap link_priv → read directory (verify signatures, decrypt names, unseal entries) → open children (verifying each `space_id = HKDF(name_key)`). Alias resolution is the **Resolver** — identity, not auth: `(namespace, alias) → {link_id, link_pub, bound_at, proof}` — with self-signed binding proofs, mandatory client-side TOFU pinning (hard-warn on key change), and key transparency as the systemic roadmap fix; the protocol and a reference resolver (alias rows stored through the kernel itself) are OSS, the hosted resolver with verified alias claiming is commercial trust infrastructure. The trust architecture reads as an assembly line — identity mints bindings → auth mints tokens → reachability gates bytes — with readability (keys) running parallel, touching identity at exactly one object, link_pub; the compromise asymmetry is the threat model's spine: auth compromise leaks reachability to ciphertext; resolver compromise redirects keys (TOFU pins bound the blast radius to first contact).

**Lifecycle.** Local genesis, account linking, key backup, and initial database bootstrap belong to multilite and the future identity layer, not the current Homebase client. Cross-space setup must be an idempotent saga because there is no client-global oplog or cross-space transaction. Homebase's current `attach` stores and verifies a `SpaceEnvelope`; it does not upload a snapshot, acquire an implicit lease, or apply data.

**Code layout.** The identity layer folds into existing crates as modules, not new crates: `homebase-client` gains `homebase_client::identity` with the enforced module rule that core client modules never import it; `homebase-core` takes the binding/proof types; `homebase-wire` carries the protocol types; the server crate (`homebase`) gains a resolver module beside the kernel — the client/server crate split then enforces the security invariant that the resolver can never link custody code, and the `homebase` binary composes kernel + resolver as subcommands (supersedes the earlier planned `homebased` binary). `multilite` is the SQL crate in the same monorepo. Promotion of modules to crates waits on a tripwire: an external consumer needing identity without the client.

## Milestones — kernel first, by design

The server is no longer trivially dumb: it serializes admission, preserves an exact log, materializes point and range mutations, maintains historical aggregates, checks versions and range assertions, detects per-device forks, and enforces lease reservation conflicts. It remains value-oblivious and has zero SQLite dependency.

1. **Kernel, in-memory** — client + server crates, transport-neutral verbs, tuple encoding, exact log, range tombstones, and aggregates. *Exit: model properties for lease overlap, barriers, expiry asymmetry, version monotonicity, dense pull, range reads, and materialization.*
2. **Kernel torture** — deterministic sim: virtual time, storage faults, crashes, acknowledgement loss, lease contention, and mixed point/range histories. *Exit: replayable seeded suites clean.*
2.5. **Job-queue dogfood** — first real consumer, kernel-only. *Exit: crash-injected workers, zero double-completions, zero lost jobs.*
3. **Persistence + crypto codec** — storage trait (in-memory for the sim; slatedb behind it: local-NVMe object store dev, S3 prod); shard WAL group commit; one atomic write batch per verb (leases + counters + data uniformly); SIV/AEAD/HKDF codec. *Exit: kill -9 mid-batch → invariants hold; server dump = ciphertext only.*
4. **Single-writer SQL, logical** — SQLite hook capture, submit-log integration, explicit admit-log application, DDL-as-text, and bootstrap. A long database-prefix lease is a multilite policy choice, not a Homebase admission prerequisite. *Exit: an unmodified rusqlite app runs and replicas converge over randomized DML/DDL.*
5. **Witness compiler + fixpoint + shapes.** *Exit: constraint gauntlet — concurrent colliding writers, zero silent violations.*
6. **Proof point** — 3 nodes, one DB, real Axum app; latency/throughput vs. Postgres; E2EE demo. Write-up leads with the compiler (mini-paper shape).

## Client coordination scaffold

The current client coordinator is deliberately a scaffold, not yet the strongest actor interpretation of the design. Each `Client` starts one runtime-independent coordinator thread, which exclusively owns fast in-memory session state and grants FIFO workflow permits per space, plus a small blocking pool for bulk crypto. Public async operations continue to run network and `MetaStore` futures in the caller's executor task while holding the permit; consequently, operations within one space are serialized end to end, while different spaces can progress concurrently and the coordinator itself never waits on slow work. A `Space<'a>` is only a lightweight borrowed handle containing `&'a Client` and a `SpaceId`: the client does not own its `Space` handles, there is no ownership cycle, and Rust prevents a handle from outliving its client.

Data admission has two explicit verbs. `space.submit_checked(...)` and `space.submit_unchecked(...)` append durably to that space's local oplog without network I/O; `space.push()` advances only that space's remote stream. `space.push_until(seq)` bounds admission at an exact local sequence, and the `Submission` returned by `submit_*` retains the owning space so `submission.push()` can provide the common per-write wait as attribution sugar. A failed target is distinguished from a target blocked by an earlier failed submission, stalls never trim the rejected suffix, and no client-wide push silently drains unrelated spaces.

Each `(space, device)` admission stream carries a cumulative SHA-256 `DeviceChecksum` over a domain separator, the prior checksum, space and device identity, and the canonical bytes of every submitted batch. The empty stream is all zeroes. Every push presents the client's confirmed checksum; the server compares it with the checksum stored beside that device's sequence high-water before evaluating the new suffix, then atomically persists the recomputed final checksum with admission. The client likewise persists the returned checksum in the same `MetaStore` transition that trims the acknowledged oplog prefix. If an acknowledgement is lost and the server is ahead, the client folds every retained active batch through the server-reported sequence and trims only when the reconstructed checksum matches; missing, altered, reordered, or divergent same-sequence history is a fatal fork. Empty rollback-marker batches extend the chain, while retired data below `neck` does not. This detects honest-server rollback and duplicate-device histories but does not prevent a Byzantine server from equivocating to clients without an external witness. A cross-device whole-space checksum is deferred until clients can receive every canonical batch or verify an authenticated checkpoint/proof.

Scalable prefix deletion is specified separately in [`DELETE_RANGE_DESIGN.md`](./DELETE_RANGE_DESIGN.md). It is intentionally not modeled as a compressed loop of point deletes: ancestor tombstones affect descendant reads, descendant tombstones affect ancestor history, and exact counts require lazy subtree-reset semantics. Homebase is an ops/log service, not a managed replica: the server retains an exact append-only admission log, and every client space has a dense replica with durable `head <= neck <= tail` in the same `AdmissionSeq` domain. Stateful `pull()` appends the complete server-log suffix at `tail`; the application advances `neck` with `mark_applied`; trim may reclaim only through `neck`. Stateless `fetch(range, after)` supports arbitrary server range reads without changing client state. A lease is usable only when admit `neck > lease.barrier`. `unlease_checked()` and `unlease_unchecked()` remain control verbs outside both logs.

This gives simple cancellation-safe race prevention and single ownership of session state without binding the SDK to an async runtime or requiring the existing borrowed store/server APIs to become owned and `'static`. The limitation is important: the coordinator serializes the **authority** to perform durable transitions, but the caller task currently applies those transitions directly to `MetaStore`; the coordinator is therefore an arbiter rather than the literal executor of every transition. Correctness depends on all relevant paths taking the per-space permit, same-space network latency causes head-of-line blocking, nested public calls can deadlock by trying to reacquire their own permit, and the per-client OS threads, unbounded channels, primitive worker supervision, and implicit shutdown are not yet production machinery.

Evolve only when pressure earns it. Near-term hardening is an explicit internal permit/context passed to non-acquiring helpers, bounded queues and backpressure, panic propagation, graceful shutdown, metrics, and shared or injected worker infrastructure when many clients make per-client threads expensive. If same-space latency becomes material, split workflows into begin/work/complete state machines: the loop records an operation id and expected state, slow work runs without exclusive ownership, and a typed completion message is revalidated before application. If the stronger correctness boundary becomes necessary, move owned store/server handles behind a true per-space actor so workers and network tasks return results over channels and only the actor applies durable transitions; cross-space transactions would then require a separate higher-level coordinator with canonical acquisition ordering.

## Client build sequence (decided July 2026): interlocking contracts, deep implementation

The kernel rebuild landed through three interlocking contracts: **ServerHandle** (space-qualified verbs), **MetaStore** (durable client truth), and the **cipher** (privacy at submission). Contracts precede consumers; submit encrypts at ingest; all durable client state moves through atomic MetaStore transitions; transport comes after in-process semantics; and higher-level SQL and identity layers compose above these boundaries.

1. **Storage vocabulary → core** — `OrderedStore`/`WriteBatch`/`MemoryStore`/conformance move to `homebase-core`; server re-exports. *Exit: workspace green, zero behavior change.*
2. **ServerHandle contract** — space-qualified verbs; closure blanket impl (in-process = `|id| server.space(id)`); uninhabited `Offline`. *Exit: verb conformance over an in-process server.*
3. **Client meta schema** — record layout (device id; per-space submit/admit cursors and logs, version high-water, leases, codec cache) plus atomic transitions over the shared store. *Exit: recomputation-audit oracle — the client twin of the server's `check`.*
4. **Engine, write-through** — device_seq/replay fence, HeldLease local deadlines, subset renew, blind-retry rules — durable truth in the schema from day one, in-memory view only. *Exit: two-clock expiry, exactly-once replay-fence, and crashy-resume tortures with fault injection on both stores.*
5. **Job-queue dogfood (rung 2.5)** — first consumer, engine tier, no crypto. *Exit: crash-injected workers, zero double-completions, zero lost jobs; verdict on the storeless tier and `put_once`.*
6. **Codec** — path-chained SIV names, XChaCha detached Seals (`cipher_epoch` = 0 in v1; scheme/op/key/write context in AAD), namespace enums, `SpaceEnvelope` + `space_id = HKDF(name_key)` verify-at-open. Pure. *Exit: Set/Delete roundtrip, tamper/wrong-key/op-kind gauntlet; prefix-correspondence property.*
7. **Replica write path** — `submit_checked` / `submit_unchecked` encode and append locally; `push`, `push_until`, and `Submission::push` drive admission; caller-directed rollback appends an empty wire batch. *Exit: exactly-once through the codec under crash and acknowledgement-loss torture; ciphertext-only-at-rest oracle.*
8. **Inbound log path** — dense authenticated `pull` appends admit batches, applications advance `neck` after durable apply, and stateless `fetch` leaves MetaStore unchanged. *Exit: paging, crash, tamper, and equivalence torture across both logs.*
9. **Client facade** — `open`/`open_offline` (store-owned device identity), `space`/`create_space` (local genesis), `replicas`. *Exit: multi-space isolation; offline-replica physics.*
10. **DiskStore** — redb-backed, directory-tied `OrderedStore`. *Exit: the shared storage conformance suite + kill-9 durability test.* (Slot-flexible: anywhere after 3.)
11. **Identity primitives** — Ed25519 link keys (X25519 sealing key derived), `LinkID` commitment, Argon2id KEK wrap with method discriminant, sign/seal. Pure. *Exit: test vectors + property tests.*
12. **Link-space formats** — root envelope record, directory codec (SIV names under a link_priv-derived key; entries sealed to link_pub *and* owner-signed), sender-signed inbox invites. *Exit: forged/tampered/replayed-entry rejection gauntlet — the auth-compromise key-substitution attack must fail in tests.*
13. **Linking saga + bootstrap** — local genesis → link (keys-before-data, idempotent, resumable), new-device chain (LinkID-direct), attach-without-link. *Exit: saga crash-torture at every step boundary; dangling-entry-never-unreadable-ciphertext oracle.*
14. **Wire re-land** — kernel proto + DTO conversions + the error-plane rule. *Exit: encoded roundtrip suite.* (Slot-flexible: re-lands as built.)
15. **Transport adapters** — HTTP service over the kernel plus a remote `ServerHandle`. *Exit: client suites re-run over loopback transport, observationally identical to in-process.*
16. **Resolver (reference)** — binding/proof types in core, resolver proto, server resolver module (alias rows stored through the kernel), client TOFU pinning. *Exit: bootstrap's resolve leg live; pin-change hard-warn tests.*

## Deliberate exclusions

No read→write lease upgrade (unlease and reacquire). No lease stealing or fencing epochs. No separate `sync` verb: submit/push owns outgoing progress, dense pull owns incoming replication, and stateless fetch owns arbitrary range observations. No managed client KV application layer. No server-side search or plaintext aggregation. No cross-space transactions or client-global oplog. No optimistic CAS/range-CAS yet. The exact admission log is replay history, not MVCC: pinned historical snapshots and server-log GC require authenticated checkpoints and explicit retention semantics. No merge engine/CRDTs and no server push channel. Byzantine-server completeness and fork consistency require an external witness or proof layer and remain out of scope.

## Open decisions

**Wire protocol — superseded (July 2026): HTTP, not gRPC.** The earlier gRPC/tonic decision is reversed by the proxied-binary topology (see "OSS boundary, topologies, and wire protocol"): plain HTTP/1.1+2 that stock middleware can route, inspect, scope-check, and rate-limit. What carries forward unchanged from the gRPC design: the **error-plane rule** (kernel rejections travel in response payloads; transport statuses are reserved for the transport plane — the `SpaceError::Kernel` vs `Unavailable` split the client retry contract is built on), one client verb trait over multiple endpoint families (now in-process, HTTP, S3-read-only), durations-cross-as-ms, and timestamps-never-cross (asymmetric expiry). The tonic protos and DTO conversions built pre-rebuild remain in the quarry as reference; the wire batch re-lands as the HTTP spec instead, including the normative middleware-legible key-prefix encoding.

**TODO — resharding mechanism (deferred; settle before wire formats freeze).** Goal: start with few shards, add more under load, no stop-the-world rehashing. Two candidate architectures, decision deferred:
(1) **Generational addressing** — shard id is the triple `(gen, num_shards, shard_idx)`: `gen` a monotonic reshard generation, `num_shards` that generation's count (power of two, so placement is `hash(space) mod num_shards` and gen-to-gen ownership is a bit-prefix refinement), `shard_idx` the slot. Scaling = bump the generation, stack the new gen's SST/WAL lineage *in front*; reads consult newer gens first and fall back through older ones until offline compaction migrates data forward and retires them — LSM semantics lifted to fleet level. Costs: read amplification across live generations; a space can't actually straddle generations (counters/leases/aggregates need read-your-writes), so each space needs a whole-space cutover anyway.
(2) **Directory with default placement** — exploit that the space is the natural movement quantum: no verb spans spaces, and the space id is the first key component, so a space is one contiguous byte range in shard storage. Placement = small directory (space → shard) holding only exceptions, default `hash(space) mod num_shards` at the current generation; growing = bump the default and migrate spaces wholesale (lazily on first touch or by background sweep); hot spaces movable individually by directory entry.
Leaning (2) — it drops the multi-generation read path and adds per-space placement freedom — but either way the shard-id *type* should carry the triple from day one so wire formats and manifests never migrate. Fits post-v1 next to space→shard placement/rebalancing (slatedb manifest views may give O(1) key-range splits for the copy step).

**TODO — Merkle-like per-prefix hashes.** Consider extending the write-time per-prefix aggregates with a subtree content hash per `(depth, prefix)` (over ciphertext + tags, so the server can maintain it without plaintext): each put updates the hash chain along the key's prefix path in the same atomic batch, exactly like the max-seq/live-count aggregates. Would buy replicas O(log n) divergence detection and anti-entropy (compare subtree hashes, descend only where they differ — cheap partial-replica verification after long offline), and strengthen fork detection beyond generation-boundary anchors toward continuous subtree attestation. Costs to weigh: hash-update amplification per write (one hash per prefix level), the choice of structure (true Merkle child-composition vs. cheaper order-insensitive incremental hash — set-homomorphic hashes allow O(1) updates but weaker guarantees), and that server-maintained hashes are integrity hints for replicas, not trust anchors, unless clients can recompute them (which they can, over ciphertext, for prefixes they fully replicate).
